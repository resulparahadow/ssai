Multi-agent section-by-section audit of js/doctrine.js DEFAULT_TRAINING. Goal: cut cached tokens (read every message) without losing behavioral 'sauce'. Read-only analysis — edits need Francesco sign-off + SHA256/push ritual.

## Headline

- Tokens now ~26420 → proposed ~15905 (save ~10515, **39.8%**)

## Verdict

Hans is broadly right: a large cut is achievable, but the realistic, no-quality-loss number is ~38-40% (26,420 -> ~15,900 tokens), not the 'cut it in half' some PARTs might suggest in isolation. The reason the ceiling is ~40% and not higher: a big fraction of the doctrine is dense, code-referenced enums/flags/thresholds (PHASE NAME REFERENCE, promise_status lifecycle, wall_subtype list, posture/fork/whale state names, trust caps, sexting multiplier, time field names, the entire TOS banned-word list) that the validators, detectors, state machine, and ToS scanner read literally — that text is incompressible by definition. The genuine fat is concentrated and mostly LOW-risk: positive REFERENCE-EXAMPLE menus (which the codebase already treats as parroting vectors and has a standing policy to replace with principle + hard-NOs), the bilingual duplication in PART 11, and repetitive WHY-IT-MATTERS / FAILURE-MODE / 'ladder breathes' narrative prose. Cutting those actively IMPROVES quality (fewer parroting vectors, one fixed self-contradiction), so for those the 'compression vs quality' tradeoff runs the right way. Two honest caveats: (1) the single biggest line item — PART 11's Spanish removal (~1,800 tok) — is a deliberate authoring decision, not a pure cleanup, so the safe-and-uncontested figure without it is closer to ~33% (~8,700 tok); with manager sign-off on Spanish removal you reach ~40%. (2) Example removal's effect on the LLM's voice is probabilistic and the Node harness can't see it — these must be live-tested in batches, because the examples, while parroting vectors, are also currently doing some 'shape' teaching. Net: yes, trim aggressively, but do it PART-by-PART with author coordination on the doctrine-judgment cuts, keep every engineering token and every NEGATIVE/hard-NO example, and re-run hash + harness + a live voice matrix after each batch. Pushing much past 40% would start eroding behavioral specificity ('the sauce' — trauma styles, key-moment beats, archetype cues) rather than removing redundancy.
